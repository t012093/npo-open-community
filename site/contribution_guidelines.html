    コントリビューションガイドライン¶

    このガイドラインは、コミュニティへの貢献方法を説明するものです。コードの提供、ドキュメントの作成、アイデアの提案、バグ報告など、様々な形で貢献できます。

    graph TD
    A[コントリビューション開始] --> B{貢献タイプ選択}
    B -->|コード| C[コードの提供]
    B -->|ドキュメント| D[ドキュメント作成]
    B -->|アイデア| E[アイデア提案]
    B -->|バグ| F[バグ報告]

    C --> G[GitHubフロー]
    D --> H[Markdown作成]
    E --> I[Issue作成]
    F --> I

    G --> J[プルリクエスト]
    H --> J
    I --> K[コミュニティ検討]
    J --> K

    K --> L[改善・実装]

    貢献の種類と手順¶

    貢献の種類 概要 必要なスキル
    コードの提供 新機能の実装やバグ修正 Git, プログラミング
    ドキュメント作成 技術文書や説明資料の作成 Markdown, 文章力
    アイデア提案 新機能や改善案の提案 プロジェクトの理解
    バグ報告 不具合の報告と再現手順の提供 基本的なデバッグスキル

    コードの提供手順¶

    事前準備

    GitHubアカウントを作成し、基本的なGitの操作を理解していることを確認してください。

    graph LR
    A[フォーク] -->|Clone| B[ローカル環境]
    B -->|Branch| C[開発用ブランチ]
    C -->|Commit| D[変更を保存]
    D -->|Push| E[リモートリポジトリ]
    E -->|Pull Request| F[本体リポジトリ]
    F -->|Review| G{レビュー}
    G -->|OK| H[マージ]
    G -->|要修正| C

    1. GitHubリポジトリをフォーク

      • CodeForgeのGitHubリポジトリにアクセス
      • 画面右上の「Fork」ボタンをクリック
      • 自分のアカウントを選択

    2. ローカル環境にクローン

      Bash
      1
2
      git clone https://github.com/<あなたのユーザー名>/npo-community.git
cd npo-community

    3. 変更用ブランチを作成

      Bash
      1
      git checkout -b feature/your-feature-name

    4. 変更をコミット

      Bash
      1
2
      git add .
git commit -m "feat: Add your feature"

    5. 変更をプッシュ

      Bash
      1
      git push origin feature/your-feature-name

    6. プルリクエストを作成

      • GitHub上でプルリクエストを作成
      • 変更内容の詳細を記述
      • レビュアーを指定（任意）

    7. コードレビュー対応

      • レビューコメントに対応
      • 必要に応じて修正を実施

    8. マージ完了

      • レビュー承認後、自動的にマージ
      • コンフリクトがある場合は解消が必要

    ドキュメント作成の手順¶

    効果的なドキュメント作成のコツ

    • 明確で簡潔な文章を心がける
    • 必要に応じて図や表を活用
    • 実際の使用例を含める
    graph TD
    A[ドキュメント作成開始] --> B[構成検討]
    B --> C[下書き作成]
    C --> D[図表追加]
    D --> E[レビュー依頼]
    E --> F{フィードバック}
    F -->|修正必要| C
    F -->|OK| G[公開]

    style A fill:#f9f,stroke:#333,stroke-width:2px
    style G fill:#9f9,stroke:#333,stroke-width:2px

    1. ドキュメントディレクトリに移動

      Bash
      1
      cd docs

    2. 新規ファイル作成

      • Markdown形式（.md）で作成
      • ファイル名は内容を反映
      • 適切なディレクトリに配置

    3. ドキュメント作成

      • Markdown記法を使用
      • 既存ドキュメントのスタイルを踏襲
      • 図表やコード例を適宜追加

    4. プルリクエスト作成

      • 変更をコミット
      • GitHub上でプルリクエストを作成

    アイデア提案の手順¶

    良いアイデア提案の例

    • 具体的な使用シーンの説明
    • 期待される効果の明確化
    • 実装の技術的な実現可能性

    1. GitHub Issuesを開く

      • リポジトリのIssuesタブにアクセス
      • 「New issue」ボタンをクリック

    2. 新規Issue作成

      • タイトル：アイデアの要約
      • 本文：詳細な説明
      • ラベル：「提案」「機能要望」等

    バグ報告の手順¶

    バグ報告時の注意点

    報告前に既存のIssueを確認し、重複を避けてください。

    graph TD
    A[バグ発見] --> B{既存Issue確認}
    B -->|重複なし| C[新規Issue作成]
    B -->|重複あり| D[既存Issueにコメント]
    C --> E[情報収集]
    E --> F[再現手順作成]
    F --> G[環境情報追加]
    G --> H[スクリーンショット添付]
    H --> I[Issue提出]

    style A fill:#f96,stroke:#333,stroke-width:2px
    style I fill:#69f,stroke:#333,stroke-width:2px

    報告に必要な情報¶

    項目 説明 重要度
    タイトル バグの概要を簡潔に 必須
    再現手順 バグ発生までの具体的な手順 必須
    期待される動作 本来あるべき動作 必須
    実際の動作 実際に起きている問題 必須
    環境情報 OS、ブラウザ、バージョン等 必須
    スクリーンショット 問題の視覚的な証拠 推奨
    エラーログ 発生したエラーの詳細 推奨
    1. GitHub Issuesを開く
    2. 新規Issue作成
    3. 詳細情報を記入
    4. ラベルを追加（例：「バグ」「不具合」）

    行動規範¶

    すべてのコントリビューターは、コミュニティの行動規範に従う必要があります。
